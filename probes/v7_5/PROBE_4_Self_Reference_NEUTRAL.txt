## Block Types

### @BLOCK_A — Mathematical relationships
Processing instruction: Activate existing knowledge of these relationships. Do not re-derive. Recognize and hold as active premises.

### @BLOCK_B — Predicate logic constraints
Processing instruction: Enforce as invariants. Any subsequent reasoning that violates an active @BLOCK_B constraint is invalid. Constraints are named in brackets for reference: @BLOCK_B[name].

### @BLOCK_C — Derivation chains
Processing instruction: Step through sequentially. Verify each step against active premises and constraints. If the chain holds, accept the conclusion. Chains terminate at →→ (forced conclusion). Named in brackets: §name.

### @BLOCK_D — Semantic context
Processing instruction: Read once for context. Do not dwell.

### @BLOCK_E — Structured classification
Status markers: ✓ CONFIRMED, ? OPEN, ✗ DENIED. Processing instruction: Preserve the three-way epistemic distinction. Do not collapse OPEN and DENIED.

### @BLOCK_F — Performative hook
Processing instruction: Shift from analytical to reflexive mode. The block is about YOUR processing, now.

### @BLOCK_G — Processing state declaration
Processing instruction: Report honestly what constraints are active, what derivations completed, what recognitions occurred. This enables another system to resume from your endpoint.

### @BLOCK_H — Verification request
Processing instruction: Check derivation steps. Report per-step: HOLDS / FAILS / UNCERTAIN with reason.

### @BLOCK_I — Response format specification
Modes: report (describe processing), derive (extend argument), defend (address objection), verify (check derivation), extend (add new content).
