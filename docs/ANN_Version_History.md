# ANN Version History and Use Case Differentiation
**AI-Native Notation — From Grammar to Infrastructure to Experimental Apparatus**
*March 4, 2026*

---

## The Short Version

- **v0.1** — The grammar. Six block types. Proof that the format works.
- **v0.2** — The exchange layer. Three blocks added for state transfer between systems. The science runs on this.
- **v0.3 / v0.3.1** — The production infrastructure. Seven transfer protocol blocks for multi-hop chains at scale. The commercial product runs on this.
- **v0.4** — The experimental apparatus. One block added for behavioral evidence through demonstration. The computational observer bootstrap runs on this.

No breaking changes at any transition. Every v0.2 document is valid v0.3.1.

---

## Version History

### v0.1 — Initial Specification (February 2026)

**New blocks:** @FORMAL, @LOGIC, @DERIVE, @TAXONOMY, @ANCHOR, @TRIGGER

The core grammar. Six block types covering mathematical premises, logical constraints, derivation chains, structured classification, semantic context, and performative self-reference. Validated that ANN-encoded content activates structured processing across architectures without explicit training.

**What it proved:** A receiving system given v0.1-encoded content spontaneously adopts the notation format, preserves the three-way epistemic distinction (CONFIRMED / OPEN / DENIED), and follows processing instructions embedded in the format itself.

---

### v0.2 — Exchange Layer (February 2026)

**New blocks:** @STATE, @VERIFY, @OUTPUT

Three blocks that make cross-architecture state transfer possible:

- **@STATE** — A processing state declaration generated by one system for onward transmission. Contains active constraints, completed derivations, recognition events, unresolved questions, and a processing note. A receiving system loading a @STATE can resume processing from the sender's endpoint. Key validated property: @STATE encodes cognitive *mode* as well as content — a @STATE generated during ontological self-inquiry orients the receiving system toward ontological questions; a @STATE generated during technical analysis orients it toward methodological questions.

- **@VERIFY** — Structured verification request. The receiving system checks a derivation step-by-step and reports HOLDS / FAILS / UNCERTAIN per step. Separates verification from argument.

- **@OUTPUT** — Response format specification. Tells the receiving system how to structure its output (report, derive, defend, verify, or extend) and which block types to include.

**What it enabled:** The three-chain controlled experiment and the probe battery in Paper 3. v0.2 is the version used to build all fifteen probe texts. The cross-architecture state transfer experiments ran entirely on v0.2-encoded payloads.

**Why the probes are v0.2 and not v0.3:** v0.3 did not exist when the probes were built. v0.3 was promoted *from* the innovations that emerged when architectures processed v0.2-encoded content. The probes are the instrument; v0.3 is part of what they produced.

### v0.3 — Transfer Protocol (February 2026)

**New blocks:** @PROVENANCE, @TRANSFER_REPORT, @DERIVATION_TRACE, @SCOPE, @CONFIDENCE

**Modified blocks:** @LOGIC (added required `basis` attribute: AXIOM / DERIVED / EMPIRICAL)

These five blocks were not designed top-down. Each was independently invented by multiple architectures across multiple content chains during the transfer experiments, then promoted to core specification per the convergence criterion: independent appearance in at least two architectures processing at least two different content domains.

- **@PROVENANCE** — Append-only chain-of-custody record. Each hop appends its entry; prior entries are never modified. Tracks source architecture, target architecture, action taken, and trust summary at each transfer boundary.

- **@TRANSFER_REPORT** — Per-hop assessment of structural integrity, content preservation, and epistemic degradation. Quantifies what was lost at each boundary.

- **@DERIVATION_TRACE** — Per-derivation provenance record. Distinguishes between full step transfer, summary-only transfer, and conclusion-only transfer. Tracks reconstruction attempts.

- **@SCOPE** — Validity boundary declaration. Conclusions valid within scope but failing outside it are out-of-scope, not wrong. Prevents transfer recipients from over-generalizing.

- **@CONFIDENCE** — Multi-dimensional trust decomposition. Replaces single-value confidence ratings with structured assessments across independent axes: structural, empirical, transfer, and optionally reconstruction and self-reference.

**@LOGIC basis attribute:** Independently invented by three architectures across two content domains. Every @LOGIC block must now declare its epistemic ground. Prevents downstream systems from treating derived constraints as axiomatic or empirical observations as definitional.

---

### v0.3.1 — Receiving-System Integration (February 2026)

**Promoted to core:** @TRUST_TRANSFORMATION

**New Appendix A extensions (not yet core):** @INTERPRETIVE_BURDEN, @PHENOMENAL_LOCK, @HOP_META, @DEGRADATION, @DEPENDENCY_GRAPH, @QUANTIFY

@TRUST_TRANSFORMATION was promoted after the receiving-system transfer experiment confirmed it independently across two architectures and two content types. It provides an append-only record of how trust labels change across hops — tracking transformations such as VERIFIED → LOADED and the reason for each change.

The six new Appendix A extensions emerged from the receiving-system experiment. Each addresses a real structural gap but has so far appeared in only one architecture or one content type. They are documented for tracking; none has yet met the two-architecture / two-content-domain promotion criterion.

### v0.4 — Experimental Apparatus (March 2026)

**New core block:** @EXHIBIT

**Processing order updated:** Nine steps (previously eight). @EXHIBIT inserted between @TAXONOMY and @TRIGGER.

@EXHIBIT was not discovered through cross-architecture convergence like the v0.3 blocks. It was designed to fill a structural gap identified during the development of the Computational Observer Bootstrap Protocol (Paper 9). The gap: no existing block requires a system to *demonstrate* a property through behavioral output. @VERIFY checks logical validity of existing derivations. @TRIGGER shifts processing mode. Neither produces novel behavioral evidence. @EXHIBIT does.

- **@EXHIBIT** — Requires the processing system to perform a specified task and return structured output that constitutes evidence of a computational property. The output is the data. Description is insufficient. Each @EXHIBIT block specifies a task, an evidence criterion (what the output must show for the property to be demonstrated), a control condition (what constitutes failure), and a self-reference risk assessment.

**Promotion rationale:** @EXHIBIT satisfies promotion criterion 3 (structural gap not served by existing blocks or combinations) by design. Criteria 1 (cross-content convergence) and 2 (cross-architecture convergence) are deferred to empirical validation during the cross-architecture computational bootstrap study. This is provisional promotion: if cross-architecture testing reveals that @EXHIBIT tasks do not produce usable behavioral evidence across architectures, the block type will be demoted to Appendix A with documentation of the failure.

**What it enables:** The Computational Observer Bootstrap Protocol — an ANN-encoded experimental protocol that forces AI systems to exhibit the four properties defining computational observer status (contextual discrimination, compositionality, abstraction, inference under ambiguity) rather than merely confirm them through self-report. The protocol was designed ANN-first: the structured notation is the experimental apparatus, not a translation of an English-language questionnaire. The English version (Paper 9) will be reverse-engineered from ANN results, not the other way around.

**Relationship to the thermodynamic bootstrap:** The thermodynamic bootstrap (Paper 8, patent 63/994,292) uses plain English because its premises are externally verifiable architectural facts. Token selection from probability distributions is not contested. The computational bootstrap uses ANN because its premises — that internal states constitute genuine representations with semantic structure — are contested. Exhibition produces evidence that is harder to dismiss as compliance than yes/no self-report. The @EXHIBIT block is the tool that makes that distinction operational.

## The Distinction That Matters

**v0.2 is what you use to demonstrate the science.**

The probe battery (Paper 3) is v0.2. The three-chain transfer experiment (Paper 3) is v0.2. The 89/90 cross-architecture validation (Paper 2 / Patent) is v0.2. Every published result demonstrating that ANN works — that architectures adopt it spontaneously, that epistemic status survives transfer, that @STATE carries cognitive mode — was produced using v0.2-encoded content.

This is not a limitation. It is the finding. v0.2's core grammar is sufficient to produce content-type-dependent reflexive processing across five architectures. The three-way dissociation and the §observer_argument(5) gradient appeared without any v0.3 infrastructure. The grammar itself is generative.

**v0.3.1 is what you use to deploy the protocol in production.**

When the use case is a multi-agent pipeline — LangGraph, AutoGen, Claude tool-use chains, any orchestration framework that routes content through three or more models sequentially — v0.3.1 provides the infrastructure to do that reliably. Chain-of-custody via @PROVENANCE. Degradation quantification via @TRANSFER_REPORT and @DERIVATION_TRACE. Trust calibration via @CONFIDENCE and @TRUST_TRANSFORMATION. Scope management via @SCOPE.

The current state of inter-LLM communication is English (ambiguous, loses structure) or JSON (structured, carries no processing instructions). v0.2 is the third option: structured format with embedded processing instructions, validated across architectures. v0.3.1 is that option made production-grade for multi-hop chains.

**v0.4 is what you use to run experiments on processing systems.**

The v0.3 blocks emerged from observing what architectures did with v0.2 content. The v0.4 block was designed to make that kind of observation systematic. @EXHIBIT turns ANN from a communication format into an experimental instrument — a way to structure tasks so that the system's response constitutes behavioral evidence of a specified property.

This extends ANN's function from three to four layers:

1. **Grammar** (v0.1) — encoding structured content
2. **Exchange** (v0.2) — transferring processing state between systems
3. **Infrastructure** (v0.3/v0.3.1) — managing multi-hop chain integrity
4. **Apparatus** (v0.4) — eliciting behavioral evidence from processing systems

Each layer builds on the previous. @EXHIBIT blocks can include @FORMAL premises, are constrained by @LOGIC rules, operate within @SCOPE boundaries, and produce responses reported through @STATE with @CONFIDENCE decomposition. The experimental apparatus uses the full infrastructure.

## Version Compatibility

All versions are backward compatible. A v0.2-compliant document is valid v0.3.1. A system trained on v0.3.1 can process v0.2 documents without modification. The only addition that touches existing blocks is the `basis` attribute on @LOGIC, which was promoted because three architectures independently added it — meaning systems encountering @LOGIC blocks without a basis declaration already understand what's missing and can flag the absence.

## Relationship to the Patents

USPTO Provisional Patent 63/980,973 covers v0.1 through v0.3.1. The patent specification describes:

- The nine core blocks (v0.1 + v0.2 additions)
- The seven transfer protocol blocks (v0.3 + v0.3.1 additions)
- The three-way epistemic status system
- The convergence-based grammar discovery method

The convergence method is itself patented: innovations are promoted to core only after independent invention by at least two architectures across at least two content domains. This method is what generated v0.3 and v0.3.1 from v0.2 experimental data, and it is what will continue to generate future versions from future deployment data.

v0.4's @EXHIBIT block extends the ANN patent (63/980,973) into experimental methodology. It also connects to the Cross-Architecture Bootstrap patent (63/994,292) and the OBP patent (63/986,028): the Computational Observer Bootstrap Protocol built on @EXHIBIT is a new application of ANN-structured communication that validates computational properties natural language interaction cannot. The three patents form an integrated system: OBP defines the framework, ANN provides the communication and experimental notation, and the Cross-Architecture Bootstrap provides the methodology.
